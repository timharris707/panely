Purpose

Implement the Panely release/versioning workflow to match the sibling skills repo, fix the hard-coded thinking/effort level bug so UI choices reflect current local CLI capability snapshots, and close the independent-agent strict-board compatibility findings.

Problem

Panely was moving toward per-deploy GitHub Releases, which conflicts with the skills repo convention that Releases are sparse semver milestone publications. Separately, Panely still treats provider effort levels as product truth even when the installed CLI parser rejects a level, such as Claude rejecting xhigh.

Solution

Use CI-on-semver-tag for GitHub Releases, with curated CHANGELOG.md sections as the primary release-note source. Replace hard-coded effort lists with daily local CLI capability snapshots that record supported effort levels, enforcement status, evidence, and safe normalization behavior. Add a local verdict-schema gate and make strict-board posture explicit where Panely cannot enforce process-level isolation.

Independent Review Coverage

Questionables

Should Panely still keep model-level fallback lists?

Yes, but only as fallback capability assumptions when the CLI cannot be checked. The UI should label those as check-required or fallback data, not live capability truth.

Should release workflow run verification in CI before creating the release?

Not in this first implementation. The governing model says releases are cut from a merged, green main; the workflow should create the release from the tag. If GitHub branch protection is weak later, add a release workflow job that also runs npm run verify.
