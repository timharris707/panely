Implementation commit for Advisory Board compatibility closeout

Purpose

Close the remaining trust and compatibility gaps between Panely Formal Board Review and the updated Advisory Board skill. The target is not merely similar prose; it is strict artifact compatibility with advisory-board/verdict@1, more honest model controls, stronger validity math, richer provenance, and a clearer isolation posture.

Problem

Panely now has a useful Formal Board Review mode, but it still claims the Advisory Board verdict schema name without matching the schema, marks some split-seat runs valid, exposes Claude xhigh as enforceable when that may drift by CLI version, lacks several canonical artifacts, and uses the first debating seat as synthesizer. Those are exactly the areas where formal review trust can degrade.

Compatibility line: Panely must not emit a file named verdict.json with schema advisory-board/verdict@1 unless board_verdict.py accepts it.

Solution

Patch Panely’s Formal Board layer to produce validator-compatible verdicts, require at least two same seats to complete both independent and rebuttal rounds, normalize Claude reasoning honestly, emit richer run metadata and HTML handoff artifacts, and document the current isolation model. Validate against both Panely tests and the installed Advisory Board skill scripts.

  • [] Build each board entry with seat, model, lens, round_verdicts, and dropped.
  • [] Parse per-round seat verdicts from seat output, defaulting cautiously to caution only when the seat ran but omitted a verdict.

1.2 Testing Strategy

[] Phase 2: Validity, Reasoning Honesty, and Synthesis Neutrality

Fix the subtle trust bugs: same-seat round completion, Claude effort drift, and chair bias in final synthesis.

2.1 Same-Seat Validity

  • [] Update validity reason and tests for A+B in Round 1, B+C in Round 2 invalidity.

2.2 Claude Thinking Normalization

  • [] Verify current claude --help; if xhigh is unavailable, normalize xhigh -> max with a UI/API note.

2.3 Synthesis Neutrality

  • [] Use a non-debating clerk identity for synthesis; it may route through an available model, but it must not be counted as a board seat.
  • [] Build the synthesis prompt as a neutral clerk task over board artifacts, not as a continuation of any seat persona.
  • [] If the runtime must route the clerk call through a model also used by a debating seat, record that as synthesisNeutrality: model-reused and require an explicit minority-report section.
  • [] Include the synthesis producer, model, and neutrality caveat in run metadata and the decision record.
