Implementation goal for model-routing truthfulness, privacy disclosure, safety polish, build confidence, formal board protocol, structured verdict artifacts, durable local runs

Purpose

Execute the seven-item trust roadmap that came out of comparing Panely with the Advisory Board skill: make model controls honest, provider disclosure explicit, public safety stronger, builds stricter, and add a true Formal Board Review mode with structured verdict artifacts and a path toward durable local execution.

Problem

Panely is already the stronger product surface, but the Advisory Board skill is still the stronger review protocol. The app currently presents some controls as more precise than the backend enforces, has weaker provider disclosure for non-public source material, lacks a strict formal verdict artifact, and runs long sessions through route-triggered background work that is useful locally but fragile for serious reviews.

Solution

Implement the roadmap in shippable gates. Gate 1 lands the immediate trust fixes: pre-planner consent, truthful model routing, scanner hardening, public path cleanup, and build validation. Gate 2 adds a minimum viable Formal Board Review mode behind explicit protocol data: source packet hash, independent Round 1, rebuttal Round 2, seat status, and artifact storage. Gate 3 adds formal verdict artifacts and extends the existing run ledger toward durable/resumable execution without forcing a heavy queue dependency immediately.

[] Phase 1: Truthful Model and Thinking Controls

Make Panely's model controls match what the backend actually sends to each CLI. This is the first trust fix because user-visible model labels and thinking levels shape confidence in every run.

1.1. Provider-specific thinking map

  • [] Create a helper that maps generic levels to provider-specific levels and records whether a level is actually enforced, normalized, or display-only.
  • [] Remove unsupported Claude xhigh display if the installed CLI only supports low, medium, high, and max; otherwise map xhigh to the nearest supported value and disclose that normalization.
  • [] Pass Codex reasoning effort explicitly through --config model_reasoning_effort=... or the current CLI-supported equivalent.
  • [] For Gemini, either pass a verified CLI thinking setting or mark the thinking control as not CLI-enforced instead of implying it is.
  • [] Update model availability/settings payloads so the UI can show supported thinking choices and enforcement status per model.

1.2. UI alignment

  • [] Update LaunchWizard thinking selectors to use model-specific supported levels.
  • [] Add concise copy in settings/plan review that distinguishes enforced thinking, normalized thinking, and unavailable thinking.
  • [] Preserve existing sessions; do not rewrite historical metadata.

1.3. Testing Strategy

Unit-test normalization and generated CLI args, then run app-wide checks.

  • [] npm test — proves routing helpers and existing advisory logic still pass.
  • [] npx tsc --noEmit — proves type-level consistency for new model metadata.
  • [] npm run lint — proves UI/helper changes are clean.
Do not exit this phase until model controls can no longer overclaim backend behavior.

[] Phase 2: Provider Disclosure and Source Material Consent

Add explicit disclosure before any non-public topic/files are sent through Anthropic, OpenAI, or Google CLIs. This must happen before AI session planning because the planner route itself sends the topic/source material to a model.

2.1. Disclosure model

  • [] Add a provider disclosure helper that derives provider names from selected advisor model IDs.
  • [] Track source sensitivity as public, non-public, or unknown, defaulting to unknown when files are attached or the user enters arbitrary private context.
  • [] Persist disclosure acknowledgement on the session so exports can record the consent state.
  • [] Add a pre-planner guard: if sensitivity is unknown/non-public and consent is absent, do not call /api/advisory/session-plan; use a local heuristic preview until consent is granted.

2.2. Launch wizard consent

  • [] Add a clean disclosure panel immediately after the topic/source step and before AI planning when material is unknown or non-public.
  • [] Block both AI planning and launch for unknown/non-public material until the user confirms provider disclosure.
  • [] Keep copy short and direct: source material will be sent through selected local CLIs to listed providers.

2.3. Testing Strategy

Test disclosure derivation and UI launch guards; verify no prompt material is logged.

  • [] npm test — proves disclosure helper and existing tests pass.
  • [] Mocked planner call test — proves unknown/non-public material cannot call the model-backed planner before consent.
  • [] Manual browser check of New Session launch — proves the confirmation flow is understandable and not intrusive.
Do not exit this phase until non-public source material cannot start a provider run without explicit acknowledgement.

[] Phase 3: Public Safety and Build Confidence

Clean public-facing polish and make repository validation stricter so people cloning the app see a healthy project.

3.1. Safety scanner hardening

  • [] npm run build — proves clean production build without ignored type errors.

Do not exit this phase until public repo safety and build output are clean.

[] Phase 4: Formal Board Review Protocol

Add a third session mode that implements the Advisory Board protocol instead of blending it into conversational roundtable. This mode should be intentionally stricter and artifact-first.

4.1. Protocol data model

  • [] Add formal-board as a first-class mode alongside roundtable and competitive.
  • [] Define formal board phases: preflight, independent round 1, rebuttal round 2, optional convergence round 3, synthesis, complete.
  • [] Add seat status: pending, ran, degraded, dropped, including reason and model provenance.
  • [] Enforce at least two runnable seats for a valid formal board review.
  • [] Update all mode unions and compatibility surfaces: session creation, stored session normalization, decision records, briefs, archive/export, event polling, and sidebar filters. Existing sessions must keep loading.
  • [] Pin the formal protocol shape to the Advisory Board skill concepts: lens preset, cross-reading mode, source packet hash, neutral synthesis, and artifact paths.

4.2. Independent round execution

  • [] Build a shared source packet for formal runs and record its stable hash in session metadata and run metadata.
  • [] Ensure Round 1 prompts do not include other seats' outputs.
  • [] Build Round 2 board packets from Round 1 summaries or full outputs depending on context budget.
  • [] Store per-round artifacts under local untracked data, for example data/advisory/formal-runs/<sessionId>/round-1/<seat>.md.
  • [] Define concrete prompt templates for Round 1, Round 2 rebuttal, optional Round 3 convergence, and neutral synthesis.
  • [] Preserve conversational roundtable behavior separately.

4.3. UI mode presentation

  • [] Surface Formal Board Review as a distinct option when the planner infers a high-stakes review, audit, plan review, architecture gate, or launch decision.
  • [] Explain the mode in product terms, not protocol jargon: independent first pass, rebuttal, verdict artifact.
  • [] Show per-seat status during and after the run.

4.4. Testing Strategy

Unit-test protocol invariants before wiring every UI affordance.

  • [] npm test — proves formal-board invariants, existing engine logic, and decision records pass.
  • [] Mocked formal run test — proves Round 1 prompts exclude other seats' output and Round 2 prompts include only the board packet.
  • [] Manual smoke run with a short formal board topic — proves the user can see the mode and completion state.
Do not exit this phase until Formal Board Review cannot masquerade as a normal roundtable and preserves independent Round 1.

[] Phase 5: Structured Verdict and Deterministic Artifacts

Give formal runs a machine-readable outcome and a deterministic human artifact, not just conversational transcript export.

5.1. Verdict schema

  • [] Add a verdict.json builder pinned to schema: advisory-board/verdict@1.
  • [] Support gate verdicts ship, caution, and block, with confidence and rationale.
  • [] Include per-seat and per-round verdicts so the final decision can be audited.
  • [] Include evidence-backed findings, judgment calls, couldn't-verify items, dissent, risks, next actions, seat statuses, and model provenance.
  • [] Include dropped/degraded seats in the verdict; never silently omit them.
  • [] Include a minority report even on apparent consensus.

5.2. Artifact generation

  • [] Generate final-consensus.md from structured data.
  • [] Generate deterministic self-contained HTML from the same structured data or a simple local template, with no remote assets or placeholder tokens.
  • [] Validate consistency between verdict.json, final-consensus.md, and final-consensus.html.
  • [] Add export affordances for verdict JSON and formal handoff artifacts.

5.3. Testing Strategy

Test artifact consistency and schema fields, especially degraded seats.

  • [] npm test — proves verdict builder and artifact helpers pass.
  • [] node scripts/publish-safety-check.mjs — proves generated artifact paths are not accidentally publish candidates.
Do not exit this phase until final artifacts have one source of truth and no hidden seat failures.

Questionables

How large should Formal Board Review be in the first implementation?

The first implementation should be a minimum viable formal protocol: mode selection, independent Round 1, rebuttal Round 2, dropped/degraded seat accounting, and structured verdict artifacts. A full replacement conductor can come later. The goal is trustable behavior, not a sprawling rewrite.

Should Gemini thinking be hidden if no stable CLI flag exists?

Yes. If Panely cannot actually enforce Gemini thinking through the CLI, the UI should say the model supports large-context reasoning but not expose a fake thinking control. Truthful controls beat aspirational controls.

Should durable execution use a third-party queue?

Not yet. For a local-first single-user app, a persisted run ledger with resumable steps is enough. A worker process can be added after the state model is solid.

Notes

Product principle: keep Roundtable and Competitive fun and fast. Make Formal Board Review slower, stricter, and artifact-first by design.
Mode Primary Use Protocol Strength Artifact Expectation
Roundtable Exploration, advice, live discussion Conversational; later seats may see prior discussion Transcript and decision record
Competitive Ideas, pitches, critique, voting Structured competition; not a formal validity gate Top ideas, vote breakdown, transcript
Formal Board Review High-stakes plan/architecture/strategy review Independent Round 1, rebuttal, convergence, provenance verdict.json, final-consensus.md/html, run metadata

The adversarial review should explicitly check for three failure modes: privacy theater, where the planner sends source material before consent; protocol theater, where Panely says "formal" but still allows anchoring/groupthink; and durability theater, where a new ledger exists but does not actually make runs resumable.

Implementation should land in reviewable gates rather than one monolithic patch: first pre-consent and model truth, then public safety/build confidence, then Formal Board Review behind a clear mode boundary, then verdict/ledger hardening.
