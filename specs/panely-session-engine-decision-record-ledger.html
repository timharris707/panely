Purpose

Implement the next five Panely product hardening priorities: one shared session engine, a structured decision record, a persisted run ledger with recovery semantics, tighter model provenance, and a focused test harness. The goal is to reduce route drift, make artifacts more durable, and make long local-first sessions recoverable and defensible.

EngineOne state machine for turns, phases, prompts, voting, and completion.
RecordA typed output spine for recommendations, dissent, risks, and actions.
LedgerDurable step records with queued, running, done, failed, and heartbeat state.
ProvenanceRequested and observed model/source data in events and exports.
TestsHarness coverage for competitive, failure, export, and recovery flows.

Problem

Panely is now useful, but its orchestration is still fragmented. Session behavior is spread across several API routes, prompt assembly is duplicated, some routes use locks while others still do raw state mutation, and final artifacts are derived from prose and event streams rather than a durable decision object. This makes future product work riskier than it needs to be.

Solution

Introduce a small shared domain layer under src/lib/advisory/. Routes remain Next.js API handlers, but they delegate session behavior to engine, decision-record, ledger, provenance, and test helpers. Keep the migration incremental: first extract behavior without changing user-visible flow, then route more code through it, then add tests and adversarial review before commit.

Questionables

How much of the engine should be extracted in one pass?

Keep the first pass conservative. Extract shared pure helpers and integrate them with the main run route first. Manual route parity can follow if the refactor becomes too large for a safe single commit.

Should stale steps auto-resume?

Not automatically in this pass. A local CLI run may still be alive even if the UI is stale. Mark stale state clearly first; automatic requeue should require more durable process ownership or explicit user action.

Can provenance prove the exact executed model?

Only if the local CLI returns that identity. Until then, Panely should honestly record requested model, local CLI source, and verification status rather than overclaiming.
