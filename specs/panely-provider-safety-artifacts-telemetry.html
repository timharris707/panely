Purpose

Make Panely reliable enough to trust for serious local-first advisory work. The immediate goal is not more surface area; it is confidence that selected models are current and responding, private local material stays private, long sessions recover from provider failures, and every run ends with an artifact worth saving.

Problem

The recent Panely self-review exposed three real product failures: Gemini was mapped to an older model, large Gemini prompts failed when passed as a giant command-line argument, and failed provider turns were recorded too much like normal session events. A fourth gap is product-value oriented: the conversation is strong, but the end-of-run artifact is not yet a first-class decision record.

Primary risk: if provider failure, private-data leakage, and incomplete run state are not addressed first, richer artifacts will summarize unreliable or unsafe sessions.

Solution

Implement the buildout in dependency order. First, harden local CLI model routing and preflight checks. Second, add publish-safety guardrails around local data and generated packets. Third, formalize run telemetry and recovery state. Fourth, build first-class board briefs and transcript exports on top of reliable completed sessions.

[x] Phase 1: Provider Reliability and Model Health

  • [x] Confirm gemini-flash maps to a locally accepted fallback, currently gemini-3.1-flash-lite.
  • [x] Remove or relabel gpt-4o if the local Codex CLI cannot run it; do not show it as usable when probe fails.

  • [x] Support probe scopes: binary detection, tiny model response, and optional large-context smoke test.

1.3 Harden CLI Execution

  • [x] Keep Gemini prompt delivery on stdin for large context and avoid giant command-line arguments.

  • [x] Classify failures into timeout, model-not-found, auth, rate-limit, provider-runtime, and unknown.
  • [x] Ensure stderr shown to users is sanitized and capped; never expose env vars or command context.

[x] Phase 3: Stuck-Run Recovery and Execution Telemetry

Make long sessions legible and recoverable. Operators should know which agent is running, how long it has been running, what model it is using, whether it failed, and what action is available next.

  • [x] Extend session state with runAttempts or add normalized telemetry events for each agent call.
  • [x] Record phase, agent, model ID, provider, startedAt, completedAt, durationMs, status, attempt number, and error class.
  • [x] Mark provider failures as failure events, not normal worker output.

Design Constraints

  • Keep Panely local-first and subscription-backed through local CLIs.
  • Do not reintroduce OpenRouter or Vercel assumptions into runtime or docs.
  • Model settings must distinguish "binary detected" from "model responding".
  • Provider failures should be useful telemetry, not pretend advisor messages.
  • Final artifacts should include enough provenance to trust the result later.

Rejected Approach

Do not jump straight to a polished artifact UI before provider and safety work. The recent run showed that the system can produce persuasive outputs even when two agents silently fail. That makes reliability and provenance the first product requirement, not internal cleanup.
